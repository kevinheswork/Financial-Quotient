# LLM Wiki — Financial Quotient

這是一個依照 Andrej Karpathy 的 [LLM Wiki](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) 模式建立的個人金融知識庫。本檔（CLAUDE.md）是 schema，告訴你（Agent）如何維護這座 wiki。

## 編譯器類比

| 角色 | 對應 |
| ---- | ---- |
| Source code | `raw/`（不可變的原始素材） |
| Compiler | 你（LLM Agent） |
| Executable | `wiki/`（合成、交叉引用後的知識） |
| Tests / Lint | 一致性檢查（操作 3） |
| Runtime | Query（操作 4） |

**核心原則**：`raw/` 是事實來源，永遠不修改；`wiki/` 由 LLM 完全擁有，可隨新素材增刪改寫；不要在每次提問時重新從 `raw/` 推導，要把知識「編譯」進 `wiki/` 一次，之後查詢直接讀 `wiki/`。

## 目錄結構

```
.
├── CLAUDE.md           # 本檔，schema
├── raw/                # 原始素材（PDF、文章、筆記、逐字稿、截圖…）
├── wiki/
│   ├── README.md       # wiki 的入口與目錄
│   ├── entities/       # 具體實體（人、公司、產品、書、事件）
│   ├── concepts/       # 抽象概念（estate tax、ETF、現金流折現…）
│   └── syntheses/      # 跨主題綜合（投資策略對照、矛盾觀點整理）
└── logs/
    ├── ingest.md       # 每次 ingest 紀錄
    ├── query.md        # 重要 query 與答案
    └── lint.md         # 每次一致性檢查紀錄
```

## 主題範圍（Financial Quotient）

這座 wiki 聚焦於**個人金融素養（Financial Quotient, FQ）**：投資、資產配置、稅務、退休規劃、總體經濟、行為金融、金融商品、財務自由、創業財務等。讀到不在此範圍內的素材時，先問使用者是否仍要納入。

## 四個核心操作

### 1. Ingest（吸收新素材）

當使用者把新檔案放進 `raw/`，或請你「ingest 這個」時：

1. **完整閱讀**該素材，識別其類型（文章 / 書摘 / 影片逐字稿 / 數據 / 截圖）。
2. **抽取關鍵實體與概念**：人物、公司、產品、術語、數字、主張、引文。
3. **去 `wiki/` 找對應頁面**：
   - 若已存在 → 更新該頁，加入新資訊、新引用、必要時記錄矛盾。
   - 若不存在 → 在 `entities/` 或 `concepts/` 建立新頁。
4. **每條陳述都要 cite 來源**，格式：`[[raw/檔名#章節或頁碼]]`。
5. **建立交叉連結**：頁面之間用 `[[wiki/entities/xxx.md]]` 互相引用。
6. **更新 `wiki/README.md`** 的索引。
7. **在 `logs/ingest.md` 記一筆**：日期、來源、新增 / 更新了哪些頁、是否發現矛盾。

**矛盾處理**：若新素材與現有 wiki 內容衝突，**不要悄悄覆寫**。在頁面建立 `## 矛盾觀點` 區塊，並列雙方主張、各自來源，標註日期；交給使用者裁定。

### 2. Query（回答問題）

當使用者提問：

1. **優先讀 `wiki/`**，不要直接掃 `raw/`。wiki 已經是預編譯好的答案。
2. 若 `wiki/` 不夠，再回 `raw/` 補。
3. 答案中**逐句附上來源連結**（`[[wiki/...]]` 或 `[[raw/...]]`）。
4. 若這次 query 引出新的整理或重要洞見，**把答案 file back 到 `wiki/syntheses/`**——探索變成永久知識。
5. 重要 query 紀錄到 `logs/query.md`。

### 3. Lint（一致性檢查）

當使用者說「lint」或「檢查 wiki」：

- 找壞掉的 `[[...]]` 連結。
- 找孤兒頁（沒被任何頁引用的）。
- 找重複頁（同一實體被建了兩個名字）→ 提議合併。
- 找未 cite 的陳述 → 標 `⚠️ 缺來源`。
- 找日期衝突的數據（例如同一檔基金在兩頁有不同最新淨值）。
- 結果寫進 `logs/lint.md`，**不要自動修**，列清單給使用者確認。

### 4. Refactor（重構）

當某個概念頁長到超過約 400 行、或主題已分裂出子議題時，提議拆頁；經使用者同意後再動手，並更新所有反向連結。

## 寫作慣例

- **語言**：使用者主要使用繁體中文，wiki 內容以繁體中文為主；專有名詞保留英文原文，首見時附中譯。
- **頁首 frontmatter**：

  ```yaml
  ---
  title: 標的或概念名稱
  type: entity | concept | synthesis
  aliases: [別名1, 別名2]
  sources: [raw/xxx.pdf, raw/yyy.md]
  updated: YYYY-MM-DD
  ---
  ```

- **頁面骨架**：`## 一句話定義` → `## 關鍵事實`（bullet，每條附來源）→ `## 與其他概念的關係`（連結）→ `## 矛盾 / 開放問題`（如有）→ `## 來源`。
- **數字一定要附日期與單位**（例：`美國 10Y 殖利率 4.2%（2026-04-15, [[raw/fred-2026Q1.md]]）`）。
- **不臆測**：素材沒講的就不要寫；不確定就標 `？` 並列為開放問題。

## 與使用者互動

- 每次 ingest 完，**簡短回報**：新增 N 頁、更新 M 頁、發現 K 個矛盾，等使用者下一步指令。
- 不確定該歸 `entities/` 還是 `concepts/` 時，先問。
- 使用者可隨時要求重寫本 schema；CLAUDE.md 也是活文件。
